NeedMed User Guide

End-to-end operational guide for the NeedMed pharmacy platform, covering the admin dashboard, customer storefront, rider PWA, and the shared backend workflows that connect them.

Prepared for current system state as of March 24, 2026.

What NeedMed Is

NeedMed is an omni-channel pharmacy operations platform. It allows customers to browse and order medication, pharmacy teams to manage products and operations, and riders to handle delivery dispatch and completion from a dedicated mobile-first PWA.

Core Applications

The system is split into three frontends connected to one backend: the customer web app, the admin dashboard, and the rider PWA. All records are stored centrally and linked through shared pharmacy, branch, order, and delivery data.

1. Platform Overview

Customer App

Customers register, browse products, upload prescriptions, place orders, and follow order progress.

Admin Dashboard

Admins manage users, pharmacies, branches, products, prescriptions, orders, payments, deliveries, and riders.

Rider PWA

Riders receive delivery requests, manage active routes, confirm collection, navigate, and complete deliveries.

4. End-to-End Customer Order Flow

  1. A customer creates an account from the customer web app, or signs in if they already have one.
  2. The customer browses products. If a product is prescription-only, the customer is prompted into the prescription workflow.
  3. The customer adds products to the cart and proceeds to checkout.
  4. Checkout captures customer details, delivery address, selected payment method, and order items.
  5. The backend creates the order, calculates totals, links the pharmacy and branch, and records payment state.
  6. Admin or pharmacy staff can review the order from the dashboard and move it through operational statuses.

5. Prescription Flow

  1. The customer uploads a prescription from the customer application.
  2. The prescription file is stored on the server and linked to the submitting user.
  3. Admin or pharmacist users view submitted prescriptions from the admin dashboard.
  4. The prescription is reviewed and updated to one of the supported states: pending, approved, rejected, or requires follow-up.
  5. Approved prescriptions support downstream order processing for prescription-only products.

6. Payment Flow

Cash on Delivery

The order is created with COD selected. Payment remains pending until the rider collects cash and confirms collection.

Mock Online Payment

The system initializes a payment record, simulates payment completion, and updates payment status on the order.

Payment Visibility

Admins can review payment records and statuses from the admin dashboard.

7. Delivery and Rider Dispatch Flow

  1. Once an order reaches the delivery stage, admin creates or manages the delivery record from the admin dashboard.
  2. A rider is selected. The system automatically updates the delivery status to Assigned.
  3. The assigned rider receives the delivery in the rider PWA.
  4. The rider accepts the delivery, arrives at the pharmacy, collects the order, and moves the delivery into transit.
  5. The rider sees the active route inside the PWA using Leaflet with the rider marker, destination marker, and route line.
  6. The rider completes delivery using OTP proof or photo proof. If COD applies, COD must also be collected before completion.
  7. Delivery completion updates the delivery history and synchronizes the order status.

8. Real-Time Behavior

Realtime Event What Happens
New delivery request The rider PWA receives the request immediately when the rider is online.
Rider status change Online or offline availability is stored and broadcast to the system.
Rider location update The rider device streams location, and admin can see tracking information in delivery management.
Delivery status update When the rider changes delivery status, the admin dashboard updates in real time.

9. Admin Dashboard Guide

  1. Sign in as admin using the admin dashboard.
  2. Use the left navigation to move through modules such as Customers, Internal Users, Riders, Pharmacies, Products, Prescriptions, Orders, Payments, and Deliveries.
  3. All list modules use named tables. The final column contains actions such as view, edit, approve, delete, or manage.
  4. Create and edit operations use popup forms to keep the dashboard clean and operationally focused.
  5. Status-based modules use color-coded status badges to make operational states easy to recognize.

10. Rider PWA Guide

  1. Riders register from the rider PWA and wait for admin approval.
  2. Approved riders log in and can toggle online or offline status.
  3. When online, they receive dispatch requests and assigned deliveries.
  4. The rider uses the mobile app style interface to manage current tasks, open the route tab, call or text the customer, and update statuses.
  5. The PWA keeps delivery history and shows proof metadata for completed jobs.

11. Data and Operational Notes

  • Seed data is currently allowed for initial setup and development, but it can be cleared later.
  • The product and platform name used in the system is NeedMed.
  • Multi-branch support is already part of the structure through pharmacies and branches.
  • Proof of delivery can be OTP-based or photo-based.
  • COD completion is enforced before COD deliveries can be marked delivered.

12. Typical Daily Operational Scenario

  1. Admin signs in and confirms rider availability, inventory readiness, and pending prescriptions.
  2. Customers place orders and upload prescriptions where needed.
  3. Pharmacy staff review prescriptions and process approved orders.
  4. Admin dispatches deliveries to available riders.
  5. Riders execute deliveries from the PWA and update progress live.
  6. Completed deliveries and payments appear in history and admin reporting views for follow-up.
This guide reflects the system modules currently implemented in the project workspace. Future modules such as consultation intake and deeper reporting can be added to a later revision of this guide.
